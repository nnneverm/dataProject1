📌 一图看懂

本次需求建立平台统一的 数据云盘 模块，作为 AI 工程资产中心仓库，采用 工程项目 Project → 服务 Service → 版本 Version 三层结构，对齐"一个业务方向下含多个 AI 能力、每个能力按时间线持续迭代"的真实组织模式。

🎯 解决什么问题
AI 工程资产散落各产线本地，没有版本概念，难以回溯；业务上"项目"和"AI 能力"是两个粒度，需要承载这两层关系的产品形态。
💡 核心方案
以工程项目承载业务上下文（产线/负责人/可见性），以服务承载实际资产（1 个 LakeFS 仓库 = 数据 + 模型 + 代码 + 版本）。
🏗 模块组成
工程项目列表 + 项目详情 + 服务详情三个层级页面；新建流程先建项目，再在项目下建服务；服务详情含版本历史 / 数据 / 模型 / 代码四个 Tab。
🔗 原型 & 资产
原型：10.9.27.34:3030/asset-management.html
存储：LakeFS（每个服务 1 个仓库）

一、背景与目标

🎯 终极目标
  1. AI 工程部全部线下训练使用的数据集，统一在平台管理
  2. 基于沉淀的数据，探索与挖掘数据应用的场景

业务背景

平台承担多产线（行服、安全、公路算量、计价产品线、安装算量、土建、呼叫中心、斑马进度、清标等）的 AI 工程化资产管理。在实际业务中：

  • 一个业务方向（如「招投标 AI」「工地视觉巡检」）天然由多个 AI 能力服务组成（资信审查、技术标评审；安全绳系挂识别、临边防护识别…）
  • 每个服务都有独立的训练数据、模型权重、推理 / 训练代码，且迭代频繁
  • 支持网页和命令行两种数据上传方式

业务挑战

  • 缺乏统一的资产管理入口，数据集、模型和代码散落在各产线本地或对象存储中，没有版本概念，迭代后难以回溯历史
  • 业务上「工程项目」和「AI 能力服务」是天然不同的粒度，但缺少承载这两层关系的产品形态，团队不易整体协作
  • 团队成员无法在线浏览文件结构、下载特定版本或查看训练指标
  • 优质资产无法便捷共享至全平台，跨团队复用、知识沉淀效率低

目标

建立平台统一的 数据云盘 模块，作为 AI 工程资产的中心仓库，从设计之初即采用 「工程项目 → 服务」两层结构，对齐业务组织方式：

  1. 工程项目（Project）：以业务方向为粒度组织，是若干 AI 能力的集合（如「招投标 AI 套件」「工地视觉巡检」），承载业务上下文、负责人、产线归属和可见性
  2. 服务（Service）：以单个 AI 能力为粒度，每个服务对应一个独立的 LakeFS 仓库，独立管理数据、模型、代码和版本，独立发版
  3. 支持网页上传和命令行推送（lakectl）两种方式发布版本
  4. 支持按产线、关键字筛选工程项目，进入项目后查看其下所有服务
  5. 私有项目可一键发布至「共享数据」，面向全平台开放浏览与引用

二、用户故事

2.1 产线负责人 业务方向规划

  1. 作为产线负责人，我希望先创建一个「工程项目」描述业务方向（如招投标 AI 套件），再在项目下增量新建多个 AI 服务，以便业务体系清晰、可整体管理。
  2. 作为产线负责人，我希望按产线和关键字快速定位某个工程项目，进入后看到该项目下所有服务的发版状态。

2.2 算法工程师 能力开发与发版

  1. 作为算法工程师，我希望在某个工程项目下新建服务并独立发版，以便每个 AI 能力都有自己独立的数据、模型、代码与版本历史。
  2. 作为算法工程师，我希望在服务详情页查看版本历史、数据 / 模型 / 代码文件目录树，以便在需要时下载或切换到指定历史版本。
  3. 作为算法工程师，我希望通过 lakectl 命令行将版本资产直接推送到平台，融入现有 CI/CD 工作流，无需手动打包上传。

2.3 平台管理员 资产治理

  1. 作为平台管理员，我希望将优质私有项目（或其下的服务）一键发布至共享数据区，以便其他产线的成员能够浏览和引用。

⚠️
关键约束：
  • 服务名称在同一工程项目内不可重复，但不同项目之间可以重名（服务的全局唯一标识为系统自动生成的 Service ID，如 svc-A265）
  • 工程项目的「产线」「可见性」会被新建服务默认继承（可在服务层覆写）
  • 项目有项目负责人；服务有服务负责人（默认=创建者，可在「服务设置」中变更，常用于交接）；版本不记录发版人（命令行 / lakectl 上传场景下身份难以稳定获取，版本归属服务即可）
  • 接口文档地址挂在服务层（一个服务一份），不随版本变化；可在创建服务时填写或在「服务设置」中变更
  • 发版以服务为单位进行；项目本身没有版本号
  • 每个服务的版本号在服务内全局唯一且单调递增（首版 v1.0.0，由系统自动 +1 建议）；同一时刻一个服务只有一个「最新」版本
  • 历史版本只读、不可覆盖，可下载或切换为最新版（生成新 commit），保证可追溯

上传流程

  1. 选择文件后点击「开始上传」，弹窗进入上传进度状态
  2. 进度条分步展示：校验文件格式 → 解压 → 导入数据 → 处理标注 → 建立索引 → 完成
  3. 上传完成后展示结果摘要（成功导入 / 重复去除 / 已标注 / 未标注 / 格式错误 各数量）
  4. 点击「完成」关闭弹窗，版本记录立即出现在版本历史 Tab

Tab 2：命令行推送

基于 lakectl（LakeFS 官方 CLI），分 3 步操作：

Step 1 — 写入环境变量（首次，加到 ~/.bashrc 后 source 生效）

export LAKECTL_SERVER_ENDPOINT_URL=https://lakefs.cvforce.example.com
export LAKECTL_CREDENTIALS_ACCESS_KEY_ID=<your-access-key>
export LAKECTL_CREDENTIALS_SECRET_ACCESS_KEY=<your-secret-key>

# 安装 lakectl（首次）
curl -L https://lakefs.cvforce.example.com/dl/lakectl -o /usr/local/bin/lakectl && chmod +x /usr/local/bin/lakectl

Step 2 — 设置当前服务仓库路径（每个服务设置一次，Service ID 自动填入）

export REPO=lakefs://{Service ID}/main

Step 3 — 上传目录 & 提交版本（每次发版执行）

lakectl fs upload -r $REPO/data/ ./data/
lakectl fs upload -r $REPO/models/ ./models/
lakectl fs upload -r $REPO/code/ ./code/
lakectl commit $REPO -m "v1.2.0: 新增夜间场景样本500条"
💡
lakectl fs upload 只传差异对象；lakectl commit 完成后平台 Webhook 自动入库，记录出现在「版本历史」列表。

此 Tab 仅展示指令，无上传按钮（「确认发布」按钮在此 Tab 隐藏）。

六、权限设计

数据云盘围绕「创建、发版、删除」三类核心操作设计权限。所有已登录用户都能创建项目和服务、发布版本，只对「删除」做收敛：自己创建的服务自己可删，租户管理员可删任意，已共享出去的资产任何人都不能删。

6.1 角色定义

角色说明
普通成员所有已登录平台的用户
创建者创建该工程项目 / 服务的用户（记录在资产的 creator 字段）
租户管理员平台授予租户管理员角色的用户（跨项目、跨服务有管理权）

6.2 权限矩阵

操作 普通成员 创建者（自己创建的资产） 租户管理员
创建工程项目✅—✅
在项目下创建服务✅—✅
发布版本（发版）✅—✅
修改项目 / 服务设置✅✅✅
删除服务❌✅（仅自己创建的）✅（任意服务）
删除工程项目❌❌✅
删除已共享至共享数据的项目 / 服务❌❌❌

6.3 特殊约束

⚠️
共享数据保护（后续规划）：已发布到共享数据的工程项目或服务，任何角色（含租户管理员）都不能直接删除。如需删除，需先在「共享数据」中执行「下架」操作，撤回到私有状态后才能删除。
  • 项目删除前置条件：删除工程项目前，需先清空其下所有服务（或租户管理员选择"同步删除项目及其下所有服务"，需二次确认）
  • 删除操作均需二次确认：删除按钮触发后弹出确认弹窗，要求用户输入项目 / 服务名称进行二次确认
  • 删除采用软删除：服务删除后，相关 LakeFS 仓库做标记删除（不立刻物理删除），保留 30 天回收窗口；超期由后台任务清理物理资源

6.4 权限校验位置

校验时机校验内容
前端根据当前用户角色控制「删除」按钮的展示 / 灰禁：非创建者且非管理员时，删除按钮隐藏；已共享资产的删除按钮在所有角色下都隐藏
接口后端在每个写操作接口（create / publish / delete）二次校验当前用户是否具备权限；前端隐藏只为体验优化，权限以后端为准
审计所有删除操作记录到审计日志（操作人、时间、操作对象、操作前后状态），保留 6 个月
